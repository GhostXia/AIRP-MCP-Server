# Skill 与 MCP 的分工：静态指导层 vs 动态数据层

> **性质**：设计参考。给开发者 / 使用者一个「RP 数据该放进 skill 还是走 MCP」的判断框架。
> **一句话**：**skill = 静态指导**（人设 / 文风），**MCP（AIRP）= 动态数据**（状态 / 历史 / 记忆 / 关键词世界书）。两者**互补、非二选一**；完整 RP = **混合**。

---

## 1. 起因：能不能把 预设/世界书/角色卡 炼成 skill，独立 session 同载来 RP？

可以一部分，且抓住了关键杠杆（独立干净 session → 人设主导 → 治「死人化」），但**不完整**。下面拆清楚。

## 2. Agent 能同时用几个 skill？

**没有固定上限。** Skill 是**渐进披露的上下文**：平时只常驻所有 skill 的「名字 + 描述」（Token 占用少），用到才把那份正文载入。所以「同时能用几个」= **上下文窗口装得下几份正文**，受 token 预算约束，不是某个硬数字。

- 可用：几十个没问题；同时激活：几份到十几份，看体量；**3 个：毫无压力**。
- 代价：常驻 skill 越多，「名字+描述」清单越占常驻 context → 保持精简。
- RP 要**钉住 3 个常载**（非按相关性自动触发），得用支持「强制加载」的宿主。

## 3. 哪些能炼成 skill，哪些不能

| RP 数据 | 炼成 skill | 原因 |
|:--|:--|:--|
| **角色卡** | ✅ 合适 | 人设 / 描述 / 语气是**静态**的 |
| **预设** | ✅ 合适 | 文风 / 格式规则是**静态**指令 |
| **世界书** | ⚠️ 半合适 | 本是**关键词门控**（每轮只注入相关条目）。整本塞成常载 skill = 永远全开 → ① token 膨胀 ② 角色「什么都知道」、破沉浸（正是 `apply_lorebook` 要避免的）。**小书可、大书坏** |
| **活体状态**（HP/MP/EXP） | ❌ 不行 | 每轮变；skill 是死的，追不了 |
| **会话历史 / 近期上下文** | ❌ 不行 | 随轮次增长；skill 不更新 |
| **记忆 / 封卷归档** | ❌ 不行 | 有状态、跨会话；skill 做不了 |

→ **纯 skill 的 RP**：**短、无状态、小世界**的扮演 —— 行，效果不错。**长会话 / 有状态 / 大世界 / 要记忆** —— 撑不住。

## 4. 核心：静态层 vs 动态层

- **Skill = 静态指导层**：人设是谁、怎么写。author-time，不随轮次变。
- **MCP / AIRP = 动态数据层**：状态、历史、记忆、关键词世界书、持久化。run-time，逐轮变。

**最优 = 混合**：把**卡 + 预设**炼成 skill（给独立 session 一个干净 RP 上下文，人设主导、治死人化），**同时**挂 AIRP MCP 管动态数据。

这**几乎就是 `export_context_bundle` 已经在做的事**：产出**自包含的通用 markdown 上下文包** + 旁路原始数据，喂给隔离 subagent（其注释即写「按你宿主的 skill 形态包一层」）。「炼成 skill」= 在 AIRP 通用导出之上，**宿主 / 用户侧再包成 skill 格式**这一步。

## 5. 边界

AIRP **不吐宿主专属的 `SKILL.md`**（违「通用优先于特供」），只吐**通用 markdown**，由宿主包成 Claude / Cursor / 任意宿主的 skill 形状。所以「炼成 3 个 skill」是**宿主侧打包**，不是 AIRP 功能 —— 与设计律一致（见 [ROADMAP](ROADMAP.md) §0、[README](../README.md)）。

## 6. 结论

- 同时用 3 个 skill：技术上**完全可行**。
- 静态 RP 效果（文风 / 人设）：**能达成**，且是治死人化的隔离杠杆。
- 但**别丢动态层**：状态 / 历史 / 记忆 / 大世界仍要 AIRP MCP。
- **skill + AIRP 混合**才是完整 RP，不是二选一。
